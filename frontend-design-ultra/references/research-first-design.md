# Research-First Design

Use this for `/research`, `/trend`, `/premium`, `/awwward`, `/brandkit`, `/redesign`, and any substantial design when the user has not supplied a complete brand/design system.

## Research Contract

Before final design decisions, extract mechanisms from evidence:

- Domain mechanics: what users need to understand, compare, trust, buy, or operate.
- Competitor/reference mechanics: composition, type scale, section rhythm, proof placement, interaction patterns, and overused tropes.
- Design-system standards: semantics, tokens, accessibility, component behavior, and state coverage.
- UX heuristics: visibility of status, user control, consistency, error prevention, recognition over recall, and minimalist focus.
- Accessibility standards: contrast, focus, keyboard path, labels, reduced motion, and non-color-only meaning.
- Performance constraints: media weight, font loading, JavaScript budget, animation properties, and layout stability.

Minimum packet:

- page job
- audience segment
- target action
- trust friction
- proof needed
- hypothesis
- pass/fail criteria
- evidence artifacts to capture

If browsing is unavailable, state `Research unavailable` and proceed with domain heuristics, but do not pretend the result is trend-validated.

## Research Output

Return:

- Research Read: what matters in this domain now.
- Mechanics To Adapt: 3-7 reusable mechanisms.
- Mechanics To Avoid: overused, inaccessible, too expensive, or off-brand patterns.
- Design-System Implications: type, color, grid, components, motion.
- Evidence Gaps: what needs URL, screenshot, brand asset, product image, analytics, or stakeholder input.
- Confidence Tier: High, Medium, or Low per major claim.
- Change Impact Memo: what changes, why it should help, how success will be measured, and when to re-test.

## UX Evaluation Loop

Use find -> prioritize -> test:

- Find failures with heuristic review, journey walkthrough, browser evidence, and task simulation.
- Prioritize by impact, frequency, severity, business risk, and implementation cost.
- Test again after fixes with the same route/state/viewport where possible.

For actual user studies, small formative rounds are often more useful than one large late study. If statistical claims are requested, state sample/method assumptions instead of overclaiming.

## Failure Map

Track:

- issue
- evidence
- reproducibility
- affected users/tasks
- severity
- owner/file or section
- expected impact
- acceptance condition

## Source Quality Rules

- Prefer primary sources: official design systems, standards, platform docs, recognized UX research, and live product evidence.
- Use galleries/trend sites only for mechanics, not proof of usability.
- Do not copy layouts, copy, proprietary visuals, or brand-specific art direction.
- Convert references into a local grammar: tokens, grids, section jobs, state logic, and motion intent.

## Professional Baselines

Research-informed UI must satisfy:

- user need before visual novelty
- semantic design system before component styling
- evidence near claims
- readable content hierarchy
- accessible focus, contrast, labels, and states
- motion that explains or orients
- performance budget for media, fonts, JavaScript, and canvas

## Anti-Research Slop

Reject:

- moodboard-only output with no decisions
- trend name-dropping without implementation consequences
- cloning reference identity
- treating Awwwards mechanics as usability proof
- ignoring domain friction
- designing before knowing what assets exist

## Source-Informed Baselines

Use professional baselines from usability heuristics, accessibility standards, platform/design-system guidance, and ecommerce/product research. Convert them into project-specific decisions; do not quote or expose long source text.
